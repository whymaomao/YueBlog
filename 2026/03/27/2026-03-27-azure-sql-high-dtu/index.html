Azure SQL 高 DTU 排查：Log IO 饱和

Posted on 2026-03-27 In Debug

来源: Work RCA | 日期: 2026-03-27

Azure SQL WAL 机制

Azure SQL 采用 Write-Ahead Logging（WAL）：每次 INSERT/UPDATE/DELETE 不直接写数据文件，而是先写事务日志（.ldf），由后台异步刷入数据文件（.mdf）。

DTU 的计算公式：

DTU% = max(CPU%, Log IO%, Data IO%)

因此，**单独 Log IO 饱和就足以将 DTU 推到 100%**，即使 CPU 和 Data IO 均正常。
大文件分块写入时，每个 64 KB chunk 触发一次同步日志写入，N 个 chunk = N 次串行阻塞，Log IO 随文件大小线性增长。

案例：确认是 Log IO 主导

Step 1 — Azure Portal 观察

Azure Portal Query Performance Insight 显示，DTU 峰值时段 **Log IO % ≈ DTU %**，Data IO 和 CPU 均正常：

指标 峰值时段读数
DTU % ~100%
Log IO % ~100%
Data IO % < 10%
CPU % < 20%

结论：瓶颈在写操作，直接进 Query Store 排查。

Step 2 — Query Store 按 Log Bytes 排序

在问题时间窗口内执行排查 SQL，Top 5 结果（表名已脱敏）：

exec_count total_log_bytes max_log_bytes_one_exec query_text
487 68,348,408 144,524 INSERT INTO [FILE_CHUNK_STORE] ([CONTENT],[LENGTH],[OFFSET],[FILE_ID]) VALUES (@P0,@P1,@P2,@P3)
12 269,160 123,052 DELETE TOP (4999) FROM [DEPLOY_HISTORY] WHERE [DATE] <= @P0
38 122,792 5,132 UPDATE [DEPLOY_STATE] SET [STATE]=@P0,[UPDATE]=@P1 WHERE DEVICE_ID=@P2 AND RES_ID=@P3
105 108,882 6,654 UPDATE [FILE_META] SET [CDN_URI]=@P0,[SIZE]=@P1 WHERE [FILE_ID]=@P2
45 68,492 2,868 INSERT INTO [DEPLOY_HISTORY] ([ARG1],[ARG2],[AUTHOR],...) VALUES (...)

第一行高度可疑：487 次 INSERT，每次写入 ~64 KB blob，total_log_bytes 远超其他语句之和。

Step 3 — 贡献度分析

用 CTE 精确计算目标语句占整个时间窗口总 Log IO 的比例（将 @target_query_id 替换为 Step 2 中找到的 query_id）：

结果：

total_log_bytes (该语句) total_log_bytes (全部语句) pct_contribution
68,348,408 69,295,254 98.63%

单条 INSERT 贡献了 98.63% 的事务日志写入，定位完成。

如何判读 pct_contribution：

pct_contribution 判断
≥ 80% 单语句主导，优先处理该语句
40% – 80% 显著贡献，但需结合其他语句一起看
< 40% 分散型写压力，没有单一热点，考虑整体写入频率或升级 DTU

如何判断是 High Data IO 问题

DTU% = max(CPU%, Log IO%, Data IO%) 中，Data IO 代表从数据文件（.mdf）读取页面的压力。以下信号提示 Data IO 是瓶颈：

Azure Portal 信号

指标 Log IO 主导 Data IO 主导
Log IO % 接近 DTU% 远低于 DTU%
Data IO % 远低于 DTU% 接近 DTU%
CPU % 正常 可能偏高（排序、哈希）
典型操作 INSERT/UPDATE/DELETE SELECT 全表扫描、大范围读

当 Azure Portal Query Performance Insight 显示 Data IO ≈ DTU%，而 Log IO 很低，说明是读操作在消耗 DTU。

Query Store 诊断：按物理读排序

Log IO 用 avg_log_bytes_used，Data IO 改用 avg_physical_io_reads（物理页面读次数）：

典型 Data IO 高的输出特征

query_id exec_count total_physical_reads total_logical_reads query_text
1021 3 182,450 198,320 SELECT * FROM DEVICE_STATUS WHERE ...
874 120 54,300 61,200 SELECT d.*, r.* FROM DEVICE d JOIN RESOURCE r ON ...
302 1 38,900 38,900 SELECT COUNT(*) FROM EWDEPLOY_HISTO
  • total_physical_reads 高 → 数据不在 Buffer Pool，频繁从磁盘读取 → 缺索引或 Buffer Pool 太小
  • total_logical_reads 远大于 total_physical_reads → 数据在内存但读取页数过多 → 全表扫描或索引效率差
  • exec_count 低但 physical_reads 极高 → 单次查询扫描海量数据 → 缺少合适索引

常见根因

现象 根因 方向
SELECT * + 无 WHERE 索引 全表扫描，读取所有页 添加覆盖索引
JOIN 两张大表无索引列 Hash Join 需读大量页 为 JOIN 列加索引
查询返回大结果集后在应用层过滤 不必要的数据页读取 下推过滤条件到 SQL
Buffer Pool 命中率低 DTU 层级太低，内存不足 升级 DTU 或加索引减少扫描范围

关键收获

  • Log IO = DTU 时，直接排查写操作（INSERT/UPDATE/DELETE），优化 SELECT 毫无意义。

  • Query Store 是最快的诊断工具。一条 SQL 按 total_log_bytes DESC 排序，5 分钟内定位最大 Log IO 语句，无需任何应用层埋点：

  • 大文件分块写入数据库是高风险操作。每个 64 KB chunk 是一次独立的事务日志写入，Log IO = 文件大小 × chunk 数。文件大小 × locale 扇出数 = 实际写入放大倍数（如 4 MB × 12 locale = 单次请求 ~68 MB log）。

  • CDN 上传成功 ≠ DB 写入已跳过。必须逐平台验证 CDN skip 条件的覆盖范围，不同 OS 类型可能走不同分支，遗漏平台导致仍执行 DB 写入。

  • 并发上传 DTU 超线性增长。Azure SQL 的事务日志只有单个串行写入器（WRITELOG），两路并发触发写入竞争，叠加行锁争用，实际 DTU 远超 2×。单用户测试严重低估并发 DTU 风险。

  • 高 DTU 必有具体触发动作。DTU 峰值时间戳对齐用户操作日志，几乎总能找到明确的触发事件。先找”谁在那个时刻做了什么”，再看 Query Store 确认 SQL 层证据。
