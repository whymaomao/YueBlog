Azure SQL 高 DTU 排查：Log IO 饱和

Posted on 2026-03-27 In Debug

来源: Work RCA | 日期: 2026-03-27

Azure SQL WAL 机制

Azure SQL 采用 Write-Ahead Logging（WAL）：每次 INSERT/UPDATE/DELETE 不直接写数据文件，而是先写事务日志（.ldf），由后台异步刷入数据文件（.mdf）。

DTU 的计算公式：

DTU% = max(CPU%, Log IO%, Data IO%)

因此，**单独 Log IO 饱和就足以将 DTU 推到 100%**，即使 CPU 和 Data IO 均正常。
大文件分块写入时，每个 64 KB chunk 触发一次同步日志写入，N 个 chunk = N 次串行阻塞，Log IO 随文件大小线性增长。

案例：确认是 Log IO 主导

Step 1 — Azure Portal 观察

Azure Portal Query Performance Insight 显示，DTU 峰值时段 **Log IO % ≈ DTU %**，Data IO 和 CPU 均正常：

指标 峰值时段读数
DTU % ~100%
Log IO % ~100%
Data IO % < 10%
CPU % < 20%

结论：瓶颈在写操作，直接进 Query Store 排查。

Step 2 — Query Store 按 Log Bytes 排序

在问题时间窗口内执行排查 SQL，Top 5 结果（表名已脱敏）：

exec_count total_log_bytes max_log_bytes_one_exec query_text
487 68,348,408 144,524 INSERT INTO [FILE_CHUNK_STORE] ([CONTENT],[LENGTH],[OFFSET],[FILE_ID]) VALUES (@P0,@P1,@P2,@P3)
12 269,160 123,052 DELETE TOP (4999) FROM [DEPLOY_HISTORY] WHERE [DATE] <= @P0
38 122,792 5,132 UPDATE [DEPLOY_STATE] SET [STATE]=@P0,[UPDATE]=@P1 WHERE DEVICE_ID=@P2 AND RES_ID=@P3
105 108,882 6,654 UPDATE [FILE_META] SET [CDN_URI]=@P0,[SIZE]=@P1 WHERE [FILE_ID]=@P2
45 68,492 2,868 INSERT INTO [DEPLOY_HISTORY] ([ARG1],[ARG2],[AUTHOR],...) VALUES (...)

第一行高度可疑：487 次 INSERT，每次写入 ~64 KB blob，total_log_bytes 远超其他语句之和。

Step 3 — 贡献度分析

用 CTE 精确计算目标语句占整个时间窗口总 Log IO 的比例（将 @target_query_id 替换为 Step 2 中找到的 query_id）：

结果：

total_log_bytes (该语句) total_log_bytes (全部语句) pct_contribution
68,348,408 69,295,254 98.63%

单条 INSERT 贡献了 98.63% 的事务日志写入，定位完成。

如何判读 pct_contribution：

pct_contribution 判断
≥ 80% 单语句主导，优先处理该语句
40% – 80% 显著贡献，但需结合其他语句一起看
< 40% 分散型写压力，没有单一热点，考虑整体写入频率或升级 DTU

如何判断是 High Data IO 问题

DTU% = max(CPU%, Log IO%, Data IO%) 中，Data IO 代表从数据文件（.mdf）读取页面的压力。以下信号提示 Data IO 是瓶颈：

Azure Portal 信号

指标 Log IO 主导 Data IO 主导
Log IO % 接近 DTU% 远低于 DTU%
Data IO % 远低于 DTU% 接近 DTU%
CPU % 正常 可能偏高（排序、哈希）
典型操作 INSERT/UPDATE/DELETE SELECT 全表扫描、大范围读

当 Azure Portal Query Performance Insight 显示 Data IO ≈ DTU%，而 Log IO 很低，说明是读操作在消耗 DTU。

Query Store 诊断：按物理读排序

Log IO 用 avg_log_bytes_used，Data IO 改用 avg_physical_io_reads（物理页面读次数）：

典型 Data IO 高的输出特征

query_id exec_count total_physical_reads total_logical_reads query_text
1021 3 182,450 198,320 SELECT * FROM DEVICE_STATUS WHERE ...
874 120 54,300 61,200 SELECT d.*, r.* FROM DEVICE d JOIN RESOURCE r ON ...
302 1 38,900 38,900 SELECT COUNT(*) FROM EWDEPLOY_HISTO
  • total_physical_reads 高 → 数据不在 Buffer Pool，频繁从磁盘读取 → 缺索引或 Buffer Pool 太小
  • total_logical_reads 远大于 total_physical_reads → 数据在内存但读取页数过多 → 全表扫描或索引效率差
  • exec_count 低但 physical_reads 极高 → 单次查询扫描海量数据 → 缺少合适索引

常见根因

现象 根因 方向
SELECT * + 无 WHERE 索引 全表扫描，读取所有页 添加覆盖索引
JOIN 两张大表无索引列 Hash Join 需读大量页 为 JOIN 列加索引
查询返回大结果集后在应用层过滤 不必要的数据页读取 下推过滤条件到 SQL
Buffer Pool 命中率低 DTU 层级太低，内存不足 升级 DTU 或加索引减少扫描范围

关键收获

  • Log IO = DTU 时，直接排查写操作（INSERT/UPDATE/DELETE），优化 SELECT 毫无意义。

  • Query Store 是最快的诊断工具。一条 SQL 按 total_log_bytes DESC 排序，5 分钟内定位最大 Log IO 语句，无需任何应用层埋点：

  • 大文件分块写入数据库是高风险操作。每个 64 KB chunk 是一次独立的事务日志写入，Log IO = 文件大小 × chunk 数。文件大小 × locale 扇出数 = 实际写入放大倍数（如 4 MB × 12 locale = 单次请求 ~68 MB log）。

  • CDN 上传成功 ≠ DB 写入已跳过。必须逐平台验证 CDN skip 条件的覆盖范围，不同 OS 类型可能走不同分支，遗漏平台导致仍执行 DB 写入。

  • 并发上传 DTU 超线性增长。Azure SQL 的事务日志只有单个串行写入器（WRITELOG），两路并发触发写入竞争，叠加行锁争用，实际 DTU 远超 2×。单用户测试严重低估并发 DTU 风险。

  • 高 DTU 必有具体触发动作。DTU 峰值时间戳对齐用户操作日志，几乎总能找到明确的触发事件。先找”谁在那个时刻做了什么”，再看 Query Store 确认 SQL 层证据。

The terraform plan command is a crucial part of Terraform’s workflow. It allows you to preview the changes that Terraform will make to your infrastructure before actually applying them. Here’s a step-by-step flow of what happens when you run terraform plan:

1. Load Configuration

  • Read Configuration Files: Terraform reads the .tf and .tf.json files in your working directory to understand the desired state of your infrastructure.
  • Parse and Validate Configuration: Terraform parses the configuration files and performs initial validation to check for syntax errors or misconfigurations.

2. Load State

  • Read State File: Terraform loads the current state file (terraform.tfstate) to understand the existing state of your infrastructure. This state file contains information about the resources that Terraform manages and their current configuration.
  • Synchronize with Backend: If you’re using a remote backend (like AWS S3, Azure Blob Storage, etc.), Terraform will fetch the latest state from the backend.

3. Refresh State

  • Query Infrastructure: Terraform queries the actual state of resources in the cloud provider (e.g., Azure, AWS) to get the current state of resources that are already deployed.
  • Update State File: The state file is updated with the current state information. This step ensures that Terraform’s view of the infrastructure is accurate before generating a plan.

4. Generate Execution Plan

  • Compare Desired and Current States: Terraform compares the desired state (as defined in your configuration files) with the current state (from the state file and actual infrastructure).
  • Create Plan: Terraform generates an execution plan that details the actions required to bring the infrastructure from its current state to the desired state. This includes creating, updating, or deleting resources.

5. Output Plan

  • Display Plan: Terraform outputs the plan to the terminal, showing a summary of the changes that will be made. This includes:
    • Additions: New resources that will be created.
    • Modifications: Existing resources that will be updated.
    • Deletions: Resources that will be destroyed.
  • Plan Details: For each resource, Terraform shows the current state and the planned changes.

6. User Review

  • Review the Plan: You review the output to ensure that the changes Terraform plans to make align with your expectations. This step is crucial to prevent unintended modifications.

7. Apply Plan

  • Manual Execution: After reviewing the plan, you can execute terraform apply to apply the changes. You may also apply the plan directly from a saved plan file if you use the terraform apply <plan-file> command.

Key Points

  • No Changes to Infrastructure: The terraform plan command does not make any changes to your infrastructure. It only generates a preview of what changes will occur if you decide to apply them.
  • Plan Files: You can save the plan to a file using the -out option (terraform plan -out=plan.tfplan) and apply it later using terraform apply plan.tfplan.

By following this flow, terraform plan helps ensure that you understand the implications of your changes before they are applied, reducing the risk of unintended modifications to your infrastructure.

uptime

When issue happens, first we can use linux command uptime to check the overall status of the server.
uptime command will show the current time, how long the server has been running, how many users are currently logged on, and the system load averages for the past 1, 5, and 15 minutes.
The load average is the average number of processes that are either in a runnable or uninterruptable state.

Take the above output as an example, it shows the server is running for 32 minutes, 2 users logged on, and the load average is 14.04 for the past 1 minute, 5.77 for the past 5 minutes, and 2.29 for the past 15 minutes.
which indicates that the load in server is increasing in the past 1 minute.

Checked the CPU number of the server, it has 8 CPUs. If the load average is larger that the CPU number, it means the server is under the heavy load.
To be more specific, the server is by (14.04 - 8) *100% = 604% overloaded in the past 1 minute.

SSH public key authentication is a secure way to authenticate yourself to a remote server using asymmetric cryptography.

Work flow of SSH public key authentication

  1. Client initiates connection:
    The client (your local machine) initiates a connection to the server using SSH by specifying the hostname or IP address of the server and the username to authenticate as.

  2. Server responds with its identification:
    Upon receiving the connection request, the server responds with its identification, including its public host key. This key is used to verify the server’s identity and to establish a secure channel.

  3. Client checks server’s host key:
    The client checks whether it has seen the server’s host key before. If it has, and the key matches the one previously stored, the connection proceeds. If not, the client may prompt the user to confirm the authenticity of the server’s key. Once the key is verified, the server is added in known_hosts file in ~/.ssh directory on client machine. The known_hosts file contains the information about all the verified servers by the client.

  4. Key exchange and session establishment:
    After verifying the server’s identity, the client and server perform a key exchange to agree on encryption and authentication algorithms for the session.
    Here public key is used to encrypt data and can be freely shared. While private key that is used to decrypt data and is never shared with anyone.
    Once the key exchange is complete, a secure session is established between the client and server.

  5. Public key authentication:
    If the server is configured to allow public key authentication, it will request the client to authenticate using a public key.
    The client responds by sending a list of its public keys stored in the ~/.ssh/authorized_keys file or provided by the SSH agent.

  6. Challenge-response authentication:
    The server generates a challenge and sends it to the client.
    The client uses its private key to create a digital signature for the challenge and sends it back to the server.

  7. Verification of the signature:
    The server verifies the received signature using the corresponding public key stored in the authorized_keys file.
    If the signature verification is successful, the server grants access to the client.

  8. Establishing the SSH session:
    With successful authentication, the client gains access to the server and can interact with it securely through the established SSH session.
    Commands executed on the client are sent encrypted to the server, and responses from the server are encrypted before being sent back to the client.

  9. Session termination:
    When the session is complete, either the client or server can terminate the connection, closing the SSH session.
    This detailed workflow illustrates how SSH public key authentication works to establish a secure and encrypted connection between a client and server, allowing for secure communication and remote access.

How to write a jmeter script to generete complex request payload and parse response body

  1. To specify the java version to run the script: set java home as java8 or java17
  2. Go to the installation directory. e.g: /usr/local/Cellar/jmeter/5.6.2/libexec/bin in my mac
  3. Run the java package to launch the app
  4. Add a JSR223 Preprocessor or JSR223 PostProcessor in the script
  5. JSR223 will use groovy language to write the java scripts

How to generate the complex request and parse response payload ?

For example: our scenarios needs to implement SCEP protocol for test, we can add dependency packages in the script,
and use the library to help us implement the request payload easily.

  1. add dependency jar package into JMeter external libs directory: /usr/local/opt/jmeter/libexec/lib/ext

  2. restart Jmeter app to load the external jar packages

  3. write your own processor groovy handling functions by importing the jar packages

  4. jmeter provides some built-in functions to help us:

    • ctx: use ctx to get the response body / headers for handling

    • vars: use vars to set and get the variables and pass between the scripts

    • log: use log to print the logs in the jmeter console

      go to Jmeter app, Options –> Log Viewer to check the logs

    • SampleRequest: use SampleRequest to get the request body / headers for handling

How to extract common functions into library for jmeter

Refer to: How to Reuse Your JMeter Code with Jar Files

  1. create a new groovy project in intellij idea
  2. open project structure in intellij app, go to Modules, add the dependency jars used for the groovy classes, here I put the dependency libs in directory “libs”
  3. add a new groovy class in the new project
  4. open project structure in intellij app, go to Artifacts, select “From modules with dependencies” to build the artifact
  5. Go to intellij – Build – Build artifacts to generate the jar library
  6. Go to project directory, a new “out” directory generated with the artifacts
  7. Copy the jar into the JMeter ext library and restart JMeter app to load the jar

How to avoid breaking the test if response status code is not 200

Jmeter would mark the request as failed if the response status code is not 200, but sometimes we are expecting the other response code

  1. Add a Response Assertion to the request
  2. Set the response code to the expected code
  3. Set the response assertion to “Ignore Status” to avoid breaking the test

How to add custom assertion to the response

In addition to checking the response code, we also need to check the response body to make sure the test is correct

  1. Add a JSR223 PostProcessor for parsing the response body
  2. Extract the value in response body and check whether it is expected
  3. If not, set the request as false
  4. Add a JSR223 Assertion to the request
  5. In thread group settings, set the Action as “Stop Thread” when the assertion result is failed

How to use JSR223 sampler/processor in jmeter

JSR223 stands for “Java Specification Request 223.” It’s a standard within the Java platform that defines scripting language bindings for the Java Virtual Machine (JVM).
Use JSR223 allows to execute custom scripts in jmeter for complex scenarios

  1. JSR223 usage:
    • Dynamic generate request payloads, parameters or headers
    • Conditional logic: control the flow of execution based on runtime conditions or responses
    • Data Transformation: parse or format data retrieved from server responses or external sources
    • Custom assertions or validations
  1. Custom Jmeter library usage:
    • Add the custom library to the jmeter classpath by copying the jar file to the lib/ext directory of the jmeter installation directory
    • Restart the jmeter jar to reload the customer library
    • Use the custom library in the JSR223 sampler by importing the required classes and methods
  1. Use JSR223 sampler to create a custom request
    • Scenarios: Send request with dynamic client certificate
    • implement custom http client java class to support sending request with client certificate, extract it as a jar file
    • add the jar file to the jmeter library
    • add a new JSR223 sampler and call the custom http client class to send the request
    • Use pre-defined jmeter to set the response data to view in the result tree
    • Compile the JSR223 script by going to Tools –> Compile JSR223 Test Elements

How to debug jmeter test plan

  2. enable log viewer by going to Options -> Log Viewer
  3. Check logs in jmeter - GUI

  5. Or change the log level in the log4j2.xml file in the bin directory of jmeter for the specific class you want to debug
  6. Add a debug sampler in the test plan to print the Jmeter variables value
